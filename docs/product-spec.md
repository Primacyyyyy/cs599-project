# Product Spec · Mood Agent

## 1. 问题定义
传统音乐 App 的歌单生成依赖"关键词搜索 + 静态推荐算法"，存在三个痛点：
1. 听不懂自然语言情绪（"想要适合雨夜赶due的背景乐"无法直接表达）；
2. 冷启动差、不记得用户口味，每次都从零开始；
3. 单次请求、无多步推理，无法把"理解情绪 → 选曲 → 落地成歌单"串成闭环。

## 2. 目标用户
- 想用一句话描述心情就拿到贴合歌单的普通听众；
- 需要场景化背景乐（学习/运动/通勤）的用户。

## 3. 核心功能（MVP）
| 编号 | 功能 | 说明 |
| --- | --- | --- |
| F1 | 心情 → 歌单 | 输入自然语言心情，输出歌单名 + 曲目列表 |
| F2 | 跨会话记忆 | 记住同一用户历史心情与口味，个性化后续推荐 |
| F3 | 可选落地 Spotify | 配置后可把歌单写入真实 Spotify 账户 |
| F4 | 工具化 / MCP | 搜索与建歌单能力以 MCP 工具对外暴露 |

## 4. 改造前 vs 改造后（相对 base 仓库 mood-playlist-ai）
| 维度 | 改造前（原仓库） | 改造后（本项目） |
| --- | --- | --- |
| 控制流 | 线性单次脚本 | LangGraph 状态机 + 条件分支 |
| LLM | OpenAI 硬依赖 | DeepSeek（OpenAI 兼容，可换） |
| 记忆 | 无 | SQLite 跨会话个性化记忆 |
| 工具 | 函数内联调用 | 工具化 + MCP server 暴露 |
| 可观测 | print 调试 | 节点级结构化日志 + 耗时 |
| 可运行性 | 强依赖 Spotify OAuth | 无 Spotify 也能跑核心闭环 |
| 容错 | 报错即中断 | 工具失败降级、不中断 agent |

## 5. 非目标
- 不做音频流播放；不训练任何模型（亦规避 Spotify 关于 API 数据训练的限制）。
